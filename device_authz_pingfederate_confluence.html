1Overview

The OAuth 2.0 Device Authorization Grant RFC 8628 enables devices with limited or no browser capability to obtain an access token on behalf of a user. Common examples include smart TVs, set-top boxes, IoT sensors, CLI tools, and kiosks that cannot host a redirect-based login flow.

PingFederate supports this grant natively from version 11.1 onward and integrates cleanly with OpenID Connect (OIDC) for identity tokens and OpenID Connect Client-Initiated Backchannel Authentication (CIBA) RFC 9126 for fully decoupled push-based authentication.

1.1 When to Use This Grant Type

⚠ Not Appropriate For
Do not use this grant for web apps or mobile apps that can host a redirect URI. Use Authorization Code + PKCE for those cases. The Device Authorization Grant is strictly for constrained devices.

Use the Device Authorization Grant when all of these apply:

  • The device cannot open or redirect to a browser (TV, CLI, headless IoT).
  • The device can display a short URI and user code on-screen, via serial console, or printed label.
  • A secondary device (phone, laptop) is available for the user to complete login.
  • PingFederate is the authorization server with Device Authorization enabled.

1.2 Four-Step Flow

1
Device Registration
Device POSTs to /as/device_authz and receives a device_code, user_code, and verification_uri.
2
User Action
User visits the verification URI on a secondary device and enters the user code. PingFederate presents login (and optionally MFA and consent).
3
Device Polling
Device polls /as/token.oauth2 at the specified interval until authorization completes or the code expires.
4
Token Issuance
PingFederate returns an access_token, optionally a refresh_token, and — when OIDC scopes are requested — an id_token.

3.4 Authentication Policy

  1. Go to Authentication → Policies → OAuth Authentication Policies.
  2. Create or edit a policy scoped to Device Authorization requests.
  3. Recommended: apply MFA (PingID, TOTP) as part of the authentication chain.
  4. Map user attributes to the access token claim set as required.
✓ Best Practice
Apply a stronger authentication policy for device flows than for standard browser flows. Because the device is often unattended (e.g., a TV), enforcing MFA on the secondary device provides an additional assurance layer at no cost to the device UX.

4OIDC Configuration OIDC

When the device needs to identify the authenticated user — not just authorize API access — use OpenID Connect scopes alongside the Device Authorization Grant. PingFederate will issue an id_token in addition to the access_token.

4.1 Enable OpenID Connect

Navigate to: OAuth Settings → Authorization Server Settings

  1. Under OpenID Connect, check Enable OpenID Connect.
  2. Set the ID Token Signing Algorithm — recommended: RS256.
  3. Confirm the JWKS Endpoint is active (required for consumers to validate the id_token signature).
  4. Click Save.

4.2 Standard OIDC Scopes

ScopeClaims IncludedNotes
openidsub, iss, aud, exp, iatRequired to trigger ID token issuance.
profilename, given_name, family_name, picture, updated_atBasic user profile.
emailemail, email_verifiedUser's email address.
phonephone_number, phone_number_verifiedInclude if needed by the device use case.
offline_accessN/A — enables refresh tokensRequest this scope to receive a refresh_token.

4.3 Attribute Contract & Token Mapping

  1. Navigate to OAuth Settings → Clients → [your client] → Attribute Mapping.
  2. Map your identity store attributes (LDAP, PingDirectory) to the OIDC claim names above.
  3. Set sub to an immutable user identifier (e.g. uid or objectGUID) — never a mutable attribute like email.
  4. Set the ID Token Lifetime (e.g. 3600 seconds).

4.4 OIDC Device Authorization Request

POST /as/device_authz — with OIDC scopes
POST /as/device_authz HTTP/1.1
Host: auth.example.com
Content-Type: application/x-www-form-urlencoded

client_id=my-device-client
&scope=openid profile email offline_access

The token response will include an id_token alongside the access_token — see Section 7 for full examples.

4.5 Validating the ID Token

  1. Fetch the JWKS from https://auth.example.com/.well-known/jwks.json and cache it (rotate on kid mismatch).
  2. Verify the JWT signature using the public key matching the kid header claim.
  3. Check iss matches your PingFederate issuer URI.
  4. Check aud contains your client_id.
  5. Check exp is in the future (allow up to 60 s clock skew).
  6. Check auth_time and acr meet your assurance requirements.
ℹ Library Recommendation
Use a well-maintained OIDC/JWT library — never implement JWT verification manually.
Python: python-jose  |  Node.js: jose  |  Java: nimbus-jose-jwt  |  .NET: Microsoft.IdentityModel.Tokens

5CIBA / Backchannel Authentication CIBA RFC 9126

OpenID Connect Client-Initiated Backchannel Authentication (CIBA) is a related but distinct flow that eliminates the user code step entirely. Instead of displaying a code for the user to type, the client sends a login hint and PingFederate pushes an authentication request directly to the user's registered device (e.g. via PingID push notification).

⟳ Device Authorization vs CIBA
Device Authorization Grant: User sees a code on the device → visits a URL → types the code → completes login on their phone.

CIBA: Application sends a login hint (e.g. email) → user receives a push notification → approves on their phone. The constrained device never displays a code.

5.1 When to Use CIBA

  • The application knows the user's identifier (email, phone, or subject) before the authentication request.
  • The user has a registered out-of-band authenticator (e.g. PingID mobile app, FIDO2 push).
  • You want to avoid displaying a code (e.g. IVR/telephony, kiosk, call centre).
  • Your use case requires fully decoupled authentication — the consumption device and the authentication device are entirely separate.

5.2 CIBA Delivery Modes

ModeHow it WorksUse Case
PollClient polls the token endpoint until the user approves on their authenticator.Simpler to implement; suitable when push latency is unpredictable.
PingPingFederate calls a client-provided client_notification_endpoint when authentication completes. Client then fetches the token.Lower latency; requires the client to expose an HTTPS endpoint.
PushPingFederate POSTs the full token response directly to the notification endpoint.Best for server-side backends; not suitable for public clients.

5.3 Enable CIBA in PingFederate

Navigate to: OAuth Settings → Authorization Server Settings → Grant Types

  1. Check CIBA Grant (requires PF 11.3+).
  2. Configure the CIBA Backchannel Authentication Lifetime (recommended: 120 seconds).
  3. Set the Polling Interval (recommended: 5 seconds for Poll mode).
  4. Configure a CIBA Authentication Policy under Authentication → Policies — must include a push-capable adapter (e.g. PingID).
  5. Click Save.

8Code Examples

8.1 Python — Device Flow with OIDC + ID Token Validation

import time, requests
from jose import jwt  # pip install python-jose[cryptography]

PF_BASE   = 'https://auth.example.com'
CLIENT_ID = 'my-device-client'
SCOPE     = 'openid profile email offline_access'

def start_device_flow():
    resp = requests.post(f'{PF_BASE}/as/device_authz',
        data={'client_id': CLIENT_ID, 'scope': SCOPE})
    resp.raise_for_status()
    return resp.json()

def poll_for_token(device_code, interval):
    while True:
        time.sleep(interval)
        resp = requests.post(f'{PF_BASE}/as/token.oauth2', data={
            'grant_type': 'urn:ietf:params:oauth:grant-type:device_code',
            'device_code': device_code, 'client_id': CLIENT_ID})
        data = resp.json()
        if resp.status_code == 200: return data
        err = data.get('error')
        if   err == 'authorization_pending': continue
        elif err == 'slow_down':            interval += 5
        elif err in ('access_denied', 'expired_token'):
            raise RuntimeError(f'Flow failed: {err}')

def validate_id_token(id_token):
    jwks = requests.get(f'{PF_BASE}/.well-known/jwks.json').json()
    return jwt.decode(id_token, jwks,
        algorithms=['RS256'], audience=CLIENT_ID, issuer=PF_BASE)

if __name__ == '__main__':
    flow   = start_device_flow()
    print(f'Visit: {flow["verification_uri"]}')
    print(f'Code:  {flow["user_code"]}')
    tokens = poll_for_token(flow['device_code'], flow['interval'])
    claims = validate_id_token(tokens['id_token'])
    print(f'User: {claims["name"]} ({claims["email"]})')

8.2 Python — CIBA Poll Mode

import time, requests
from requests.auth import HTTPBasicAuth

PF_BASE    = 'https://auth.example.com'
CLIENT_ID  = 'my-ciba-client'
CLIENT_SEC = 's3cr3t'
AUTH       = HTTPBasicAuth(CLIENT_ID, CLIENT_SEC)

def start_ciba(login_hint, binding_message):
    resp = requests.post(f'{PF_BASE}/as/bc-auth', auth=AUTH, data={
        'scope':            'openid profile email',
        'login_hint':       login_hint,
        'binding_message':  binding_message,
        'requested_expiry': 120})
    resp.raise_for_status()
    return resp.json()

def poll_ciba(auth_req_id, interval):
    while True:
        time.sleep(interval)
        resp = requests.post(f'{PF_BASE}/as/token.oauth2', auth=AUTH, data={
            'grant_type':  'urn:openid:params:grant-type:ciba',
            'auth_req_id': auth_req_id})
        data = resp.json()
        if resp.status_code == 200: return data
        err = data.get('error')
        if   err == 'authorization_pending': continue
        elif err == 'slow_down':            interval += 5
        else: raise RuntimeError(f'CIBA error: {err}')

if __name__ == '__main__':
    ciba   = start_ciba('jane.smith@example.com', 'Authorise TV login ACME-TV-7842')
    tokens = poll_ciba(ciba['auth_req_id'], ciba['interval'])
    print('Access token:', tokens['access_token'][:40])

8.3 Node.js (TypeScript) — Device Flow

import axios from 'axios';
const PF_BASE = 'https://auth.example.com', CLIENT_ID = 'my-device-client';

async function deviceFlow() {
  const { data: authz } = await axios.post(`${PF_BASE}/as/device_authz`,
    new URLSearchParams({ client_id: CLIENT_ID, scope: 'openid profile email offline_access' }));
  console.log(`Visit: ${authz.verification_uri}  Code: ${authz.user_code}`);

  let interval = authz.interval ?? 5;
  while (true) {
    await new Promise(r => setTimeout(r, interval * 1000));
    try {
      const { data } = await axios.post(`${PF_BASE}/as/token.oauth2`,
        new URLSearchParams({
          grant_type: 'urn:ietf:params:oauth:grant-type:device_code',
          device_code: authz.device_code, client_id: CLIENT_ID }));
      console.log('Token:', data.access_token.slice(0,40)); return;
    } catch (err: any) {
      const e = err.response?.data?.error;
      if (e === 'authorization_pending') continue;
      if (e === 'slow_down') { interval += 5; continue; }
      throw new Error(e);
    }
  }
}
deviceFlow().catch(console.error);

9Security Considerations

🔒 Security Review Required
All device authorization and CIBA deployments must be reviewed by the IAM Security team before go-live. Complete the IAM Onboarding Checklist (Confluence: IAM/Onboarding) as part of your design review.

9.1 Transport & Code Security

  • All endpoints must use TLS 1.2 or higher. Disable TLS 1.0/1.1 on PingFederate.
  • The verification URI must always use HTTPS — never HTTP.
  • Set device_code lifetime to the minimum practical value (300–600 s). Do not exceed 900 s.
  • Use consonants-only character set to prevent transcription errors and reduce guessing risk.
  • Apply rate limiting at your network edge (API gateway, WAF) on /as/device_authz and /as/bc-auth.

9.2 Client & Token Security

  • Scope clients to minimum necessary permissions — avoid admin-level scopes for device clients.
  • Confidential clients must rotate secrets on a regular schedule (90-day maximum).
  • Store tokens in secure, encrypted storage (TPM, secure enclave, or hardware keystore).
  • Never log access tokens or refresh tokens in plaintext.
  • Revoke tokens when a device is decommissioned or a user revokes access.

9.3 OIDC-Specific

  • Always validate id_token signatures using the JWKS endpoint — never skip signature verification.
  • Set sub to an immutable identifier. Never use mutable attributes like email as the subject.
  • Cache JWKS but rotate the cache on kid mismatch to handle key rotation gracefully.

9.4 CIBA-Specific

  • Always include a meaningful binding_message so users can confirm the request is legitimate.
  • Validate the login_hint against your user directory before sending the CIBA request to prevent user enumeration.
  • The client notification endpoint (Ping/Push modes) must validate the Authorization: Bearer client notification token.
  • Prefer private_key_jwt over client_secret_basic for production CIBA deployments.
